OCI のサービス制限とコンパートメントクォータの違い

サービス制限(Service Limits)は Oracle がテナンシ全体に対して設定する資源の上限で、コンパートメントクォータ(Compartment Quotas)は顧客の管理者がその上限の範囲内でコンパートメント単位に資源を配分する仕組みです。つまり「テナンシで使える総量を決めるのが Oracle のサービス制限、その総量を社内のチームやプロジェクトへどう割り振るかを決めるのがクォータ」という役割分担です。サービス制限がテナンシレベルにあるのは、Oracle がリージョン/可用性ドメイン(AD)の物理キャパシティを計画・確保し、利用者を想定外の消費とコストから守るためです。

2 つの仕組みの比較

観点サービス制限コンパートメントクォータ
設定する人Oracle(テナンシ作成時に既定値が付与)顧客のテナンシ管理者
適用範囲テナンシ全体(制限ごとにリージョン単位・AD 単位・リソース単位のスコープがある)指定したコンパートメント(リージョン条件も付けられる)
主な目的Oracle 側のキャパシティ確保と、想定外の大量消費・コストからの保護テナンシ内での資源配分・ガバナンス・コスト統制
変更方法コンソールから制限引き上げをリクエストし、Oracle が承認クォータポリシー文を管理者が自分で作成・編集(即時反映)
上下関係クォータはサービス制限を超えられない(制限が天井、クォータはその配分)

サービス制限とは — Oracle が決めるテナンシの天井

サービス制限は、テナンシで作成できるリソース量の上限(リソース許容量)で、テナンシへのサインアップ時に Oracle によって既定値が設定されます。トライアルや Free Tier のテナンシには小さめの制限が付き、営業経由の契約では契約内容に応じた制限が設定されます。制限ごとにスコープが決まっており、リージョン単位のもの、AD 単位のもの(例: Database 系)、リソース単位のものがあります。あるスコープで制限に達しても、別の AD などほかのスコープには空きが残っている場合があります。

現在の制限値と使用量はコンソールの「Limits, Quotas and Usage(制限、クォータおよび使用状況)」ページか、Limits API / CLI で確認できます。

# テナンシの Compute サービス制限の一覧を表示(テナンシ OCID はダミー値)
oci limits value list \
  --compartment-id ocid1.tenancy.oc1..example \
  --service-name compute

制限を超えるリソース作成はエラーになります。不足する場合はコンソールから引き上げリクエストを送り、Oracle の承認後に反映されます。クォータと違い、顧客側で直接値を書き換えることはできません。

コンパートメントクォータとは — 管理者が決める配分ルール

コンパートメントクォータは、テナンシ管理者が IAM ポリシーに似た宣言的なポリシー文で作成する、コンパートメント単位の資源上限です。キーワードは set(上限を設定)、unset(既定に戻す)、zero(利用を禁止)の 3 つです。

# 開発用コンパートメントの E4 標準コアを 20 に制限
set compute-core quota standard-e4-core-count to 20 in compartment Dev

# 検証用コンパートメントでは Compute コアを一切使わせない
zero compute-core quotas in compartment Sandbox

# ホワイトリスト方式: いったん全部ゼロにして、必要なものだけ許可
zero compute-core quotas in tenancy
set compute-core quota standard-e4-core-count to 240 in tenancy

# リージョン条件付き: 東京リージョンのみ 40 コアまで
set compute-core quota standard-e4-core-count to 40
  in compartment Dev where request.region = 'ap-tokyo-1'

クォータはあくまで「テナンシに与えられた総量(サービス制限)を内側で切り分ける」仕組みなので、サービス制限を超える値を設定しても実際にはサービス制限までしか使えません。逆に zero を使えば、契約上は使えるサービスを特定のコンパートメントで意図的に封じる、といったガバナンス統制ができます。

なぜサービス制限は「テナンシレベル」にあるのか

公式ドキュメントは、サービス制限の役割を「想定外の消費急増とコストから利用者を守ること」と「ワークロードに必要な資源量を Oracle に伝え、必要なときに資源が確実に使えるようにすること」と説明しています。これがテナンシレベルである理由は、次のように整理できます。

  • 物理キャパシティの計画単位が「契約=テナンシ」だから: リージョンや AD のサーバー・ストレージは有限の物理資産で、Oracle はテナンシごとに宣言された上限をもとにデータセンターの容量を計画します。コンパートメントは顧客が論理的に自由に作れる箱にすぎず、Oracle 側の容量計画の単位にはなり得ません。だからこそ、内側の配分は顧客自身がクォータで行う設計になっています。
  • 請求と保護の単位もテナンシだから: 認証情報の漏えいやスクリプトの暴走でリソースが大量作成されても、テナンシの天井で被害(コスト)が頭打ちになります。トライアルテナンシの制限が特に小さいのも、この保護と不正利用対策のためです。
  • 可用性設計の前提になるから: ベストプラクティスでは、ピーク時の必要量に加えて「AD 障害時に別の AD へフェイルオーバーできるだけの余裕」を制限として確保しておくことが推奨されています。これもテナンシ全体で見積もる話です。

具体例を挙げます。ある会社が新サービスのローンチに向けて、東京リージョンで E4 標準コアを 500 コア使う計画を立てたとします。テナンシの既定制限が 100 コアのままだと、ローンチ当日に 101 コア目のインスタンス作成が制限超過エラーで失敗します。事前に制限引き上げをリクエストしておけば、Oracle は東京リージョンの AD にそのぶんの物理キャパシティを見込んで確保できるため、必要なときに確実にプロビジョニングできます。引き上げ後は、管理者がクォータで「本番コンパートメントに 400、開発に 80、検証は 0」のように内訳を統制する、というのが両者を組み合わせた典型的な運用です。
