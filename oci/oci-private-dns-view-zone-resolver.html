OCI プライベート DNS のビュー・ゾーン・リゾルバ三層構造

OCI プライベート DNS は「ゾーン → ビュー → リゾルバ」という三層構造を持ちます。ゾーンは DNS レコードの集まり、ビューはゾーンをまとめる論理コンテナ、リゾルバは VCN ごとに 1 つ自動作成される問い合わせエンジンです。AWS Route53 のプライベートホストゾーンが VPC に直接紐づく平坦な構造なのに対し、OCI は「ビュー」という中間層を持つことで、同じビューを複数 VCN から共有参照したり、リゾルバが複数のビューを優先順で検索するスプリットホライズン DNS を実現しています。

三つのコンポーネントの役割

プライベートゾーン (Private Zone)

通常の DNS ゾーンと同じく、ドメイン名に紐づく A / AAAA / CNAME / PTR などのレコードを格納します。ゾーン名はプライベートアドレスを指すカスタム名(app.example.internal など)でも、実在するパブリックドメイン名でも構いません。応答は VCN 内からしか届きません。

プライベートビュー (Private View)

ゾーンをグループ化する名前付きコンテナです。同じゾーン名をビューをまたいで使うことはできますが、同一ビュー内ではゾーン名は一意でなければなりません。ビューは 1 つのオブジェクトとして複数のリゾルバから参照できるため、DNS データの共有単位になります。

VCN を作成すると、システムが管理するデフォルトビューが自動生成されます。デフォルトビューは保護されており、*.oraclevcn.com 形式の VCN 内ホスト名ゾーンが含まれます。このゾーンは削除できません。

プライベートリゾルバ (Private Resolver)

VCN に 1 つ割り当てられる DNS サーバーです。「Use DNS hostnames in this VCN」を有効にした VCN を作成すると自動生成されます。リゾルバは受け取ったクエリに対して次の順番で応答を試みます。

  1. カスタムビューを登録順に検索 — リゾルバに明示的にアタッチしたビューを順番に確認
  2. デフォルトビューを検索 — 保護済みのシステム管理ゾーンを確認
  3. 転送ルール(Forwarding Rule)を適用 — 条件が一致すれば外部 DNS / オンプレ DNS へ転送
  4. インターネット DNS へ解決 — どれにも該当しない場合

「ビュー」が生む二つの利点

① 複数 VCN 間でのゾーン共有

AWS Route53 では、プライベートホストゾーンを複数 VPC と関連付けることはできますが、ゾーン自体を「グループ」としてまとめる仕組みはありません。OCI では一つのビューを複数リゾルバ(= 複数 VCN)から参照できるため、たとえば共通のサービス名解決ゾーンを一元管理できます。

# 例: 共有ビューを 2 つの VCN のリゾルバにアタッチする (CLI)
oci dns resolver update \
  --resolver-id  \
  --attached-views '[{"viewId":""}]'

oci dns resolver update \
  --resolver-id  \
  --attached-views '[{"viewId":""}]'

② スプリットホライズン DNS

同じゾーン名(api.example.internal)を異なるビューに作成し、VCN ごとに異なるビューを参照させることで、同じ FQDN に対して VCN によって異なる IP を返す構成が可能です。例えば Dev VCN のリゾルバはビュー Dev を参照し、Prod VCN のリゾルバはビュー Prod を参照する、という分離ができます。

構成イメージ

AWS Route53 プライベートホストゾーンとの比較

観点 AWS Route53 PHZ OCI プライベート DNS
構造 ゾーン → VPC (直接紐づき) ゾーン → ビュー → リゾルバ → VCN
ゾーンの共有 1 ゾーンを複数 VPC に関連付け可 1 ビューを複数 VCN のリゾルバが参照可
スプリットホライズン 同名の PHZ を別々に作成し VPC ごとに関連付け 同名ゾーンを別ビューに置き、リゾルバごとに参照ビューを切り替え
システム管理ゾーン Route53 Resolver が暗黙的に処理 デフォルトビューの保護済みゾーンとして明示的に存在
解決の優先順位 PHZ が Resolver より優先 (関連付けた VPC 内) カスタムビュー → デフォルトビュー → 転送ルール → インターネット の順

よくある活用パターン

  • マルチ VCN での共通サービス名解決 — db.shared.internal などを 1 つのビューで管理し、複数 VCN のリゾルバから参照する。
  • オンプレミスとの名前解決統合 — 転送ルールを使い、オンプレの DNS ゾーンはオンプレ DNS サーバーへ条件付き転送する。
  • 開発・本番の DNS 分離 — 同名のゾーンを Dev ビューと Prod ビューに別々に作成し、環境ごとのリゾルバに割り当てる。
