OKE のオンデマンド仮想ノード・サイクリングとは

OCI Kubernetes Engine (OKE) の仮想ノード(Virtual Nodes)でも、これまで管理対象ノード(Managed Nodes)だけが対象だった「オンデマンドのノード・サイクリング」がサポートされました。仮想ノード・プールの更新 API(UpdateVirtualNodePool)で isVirtualNodeCyclingEnabled を有効にして更新すると、taint やラベル、NSG、Pod 構成などの変更が、既存の仮想ノードを新しい構成のノードへ自動的に順次置き換える形で反映されます。置き換えの並行度は maximumSurge(既定 1)と maximumUnavailable(既定 0)で制御でき、アプリケーションの可用性とコストのバランスを調整できます。

前提: 仮想ノードとノード・サイクリングのおさらい

仮想ノードは、OKE の拡張クラスタ(Enhanced Cluster)で利用できる「サーバーレス」なワーカー・ノードです。ノードとなるインフラストラクチャの管理(パッチ適用やスケーリングなど)を Oracle 側が行うため、利用者はワーカー・ノードの OS 運用から解放されます。Kubernetes バージョンについても、コントロール・プレーンをアップグレードすると仮想ノードは自動で追従します。

一方「ノード・サイクリング」は、ノード・プールのプロパティ変更を既存ノードへ行き渡らせるために、新しい構成のノードを作成してワークロードを移し、古いノードを終了する、というインプレース更新の仕組みです。従来この機能は管理対象ノード・プール専用で、後にリブートやブート・ボリューム置換にも拡張されてきましたが、仮想ノード・プールでは使えませんでした。今回のリリースで、この「オンデマンドで任意のタイミングにサイクリングを実行できる」仕組みが仮想ノードにも広がった、というのが発表の要点です。

何ができるようになったか

仮想ノード・プールの更新時にサイクリングを有効化すると、プール内の既存の仮想ノードが新しい構成の仮想ノードへ自動的に置き換えられます。UpdateVirtualNodePool で変更できるプロパティは次のとおりで(OCI 公式 SDK / CLI の定義から確認)、これらの変更を既存ノードへ反映する手段としてサイクリングを利用できます。

  • 表示名(displayName)、サイズ(size)
  • 初期ラベル(initialVirtualNodeLabels)と taint(taints)
  • 配置構成(placementConfigurations: 可用性ドメイン/フォルト・ドメイン/サブネット)
  • ネットワーク・セキュリティ・グループ(nsgIds)
  • Pod 構成(podConfiguration: Pod 用サブネット・NSG・shape)
  • タグ(freeformTags / definedTags / virtualNodeTags)

サイクリングを制御する 3 つのパラメータ

サイクリングの挙動は、仮想ノード・プールの virtualNodePoolCyclingDetails に指定する次の 3 つで制御します(値は OCI 公式 SDK の API 定義で確認したものです)。

  • isVirtualNodeCyclingEnabled: サイクリングを有効にするかどうか。有効にして更新すると、変更を反映するために既存の仮想ノードが置き換えられます
  • maximumSurge: サイクリング中に一時的に追加してよい仮想ノード数(整数またはプール・サイズに対する割合。既定は 1)。大きくするほど並行して置き換えられ所要時間は短くなりますが、その分一時的なコストが増えます
  • maximumUnavailable: サイクリング中に同時に停止してよい既存仮想ノード数(整数または割合。既定は 0)。大きくするほどコストをかけずに並行処理できますが、可用性への影響が大きくなります

使い方(OCI CLI の例)

OCI CLI では oci ce virtual-node-pool update の --virtual-node-pool-cycling-details オプションに JSON で指定します。次の例は、taint の変更をサイクリングで既存ノードへ反映するイメージです(OCID はダミー値です)。

oci ce virtual-node-pool update \
  --virtual-node-pool-id ocid1.virtualnodepool.oc1..example \
  --taints '[{"key":"app","value":"batch","effect":"NoSchedule"}]' \
  --virtual-node-pool-cycling-details \
    '{"isVirtualNodeCyclingEnabled": true, "maximumSurge": "1", "maximumUnavailable": "0"}'

API では UpdateVirtualNodePool リクエストの virtualNodePoolCyclingDetails が対応します。また CreateVirtualNodePool にも同じフィールドがあるため、プール作成時にサイクリングの既定動作をあらかじめ設定しておくこともできます。

管理対象ノードのサイクリングとの違い

管理対象ノードのサイクリングでは「ノードの終了と置換」「ブート・ボリューム置換」「リブート」といった置き換え方式を選べますが、仮想ノードはインフラストラクチャが Oracle 管理のサーバーレス構成であるため、方式の選択はなく「新しい構成の仮想ノードへの置き換え」として動作します。また、管理対象ノードでは Kubernetes バージョンのアップグレード手段としてサイクリングを使いますが、仮想ノードの Kubernetes バージョンは従来どおりコントロール・プレーンのアップグレードに自動追従します。

なお、本記事の執筆時点では、リリースノート本文(リリース日を含む)と、cordon / drain の細かい挙動(PodDisruptionBudget の扱いなど)が仮想ノードでも管理対象ノードのサイクリングと完全に同一かどうかは、ネットワーク制限により一次情報で未確認です。適用前に文末のリリースノートおよび公式ドキュメントで最新の仕様を確認してください。パラメータ名・既定値・変更可能プロパティは OCI 公式 SDK / CLI(oci 2.181.1 / oci-cli)の API 定義で確認済みです。
