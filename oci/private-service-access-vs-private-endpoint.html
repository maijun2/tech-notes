OCI Private Service Access とは — Private Endpoint との違いと作成手順

Private Service Access(PSA)は、Object Storage などの OCI サービスの「サービス API 単位」で、VCN 内のサブネットに専用のプライベート IP と FQDN を持つエンドポイントを作る機能です。特定の Autonomous Database 1 台のような「リソース単位」でプライベート IP を作る Private Endpoint とは、この対象の粒度が最大の違いです。PSA を作成すると VCN のプライベート DNS が自動更新され、サービスの標準 FQDN がそのままプライベート IP に解決されるため、アプリケーションのコードや接続先 URL を一切変更せずにインターネット非経由の通信へ切り替えられます。追加料金は不要です。

Private Service Access(PSA)の概要

PSA は、VCN またはオンプレミスネットワーク内のワークロードから、パブリックインターネットを経由せずに OCI サービス(OCI API、Object Storage、Yum リポジトリ、Container Registry など)へアクセスするための仕組みです。作成すると、指定したサブネット内に専用のプライベート IP アドレスと FQDN を持つ「PSA エンドポイント」ができます。

  • PSA エンドポイントは VCN 内のもう 1 つの VNIC のように振る舞い、セキュリティリスト・Network Security Group(NSG)・Zero Trust Packet Routing(ZPR)のセキュリティ属性でアクセスを制御できます
  • 1 つの PSA エンドポイントは 1 つのサービス API 専用です。同じ VCN で複数サービスを使う場合はサービスごとに作成します
  • サブネットがどれであっても、同じ VCN 内のすべてのワークロードから利用できます
  • IPv4 のみ、または IPv4/IPv6 デュアルスタックのネットワークで利用できます
  • スループットは最大 8 Gb/秒(Object Storage は最大 25 Gb/秒)で、フォルトドメイン・可用性ドメインのフェイルオーバーが組み込まれています
  • 利用に追加料金はかかりません

Private Endpoint との違い — 「リソース単位」か「サービス API 単位」か

質問のとおり、両者の本質的な違いは対象の粒度です。あわせて、従来からある Service Gateway とも比較すると位置づけが分かりやすくなります。

観点Private Service Access(PSA)Private Endpoint(PE)Service Gateway(SGW)
対象の粒度サービス API 単位(例: リージョンの Object Storage API 全体)リソース単位(例: 特定の Autonomous Database、特定の Object Storage 名前空間)リージョンの対応サービス全体(広範)
VCN 内の実体専用のプライベート IP + FQDN(VNIC 相当)リソースごとのプライベート IP + 専用 FQDNルートターゲットのみ(IP は持たない)
DNS / 接続先 URLサービスの標準 FQDN がそのままプライベート IP に解決(コード変更不要)リソース専用の FQDN を使う(接続先の変更が必要な場合あり)標準 FQDN のまま(パブリック IP 宛をルートで吸収)
アクセス制御セキュリティリスト / NSG / ZPR で IP 単位に制御可能エンドポイントごとに細かく制御可能ルートとセキュリティルールのみ(粒度が粗い)
向いている場面URL を変えずにサービス単位で私設化・厳格化したい特定リソースだけに閉じたアクセスを許可したい手早く広範に私設アクセスを確保したい

つまり、「この VCN から使う Object Storage API への経路を丸ごとプライベートにしたい」なら PSA、「この Autonomous Database 1 台だけに VCN 内からのアクセスを閉じたい」なら Private Endpoint、という使い分けになります。3 つとも OCI ネットワーク内に通信を閉じ込める点は共通です。

PSA の具体的な利用メリット

  • コード変更ゼロで私設化できる: PSA 作成時に VCN のプライベート DNS リゾルバが自動更新され、対象サービスの標準 FQDN が PSA のプライベート IP に解決されます。SDK・CLI・Terraform も通常のサービス URL のまま動作します
  • サービス単位のきめ細かい制御: Service Gateway のように「リージョンの対応サービス全部」を一括で通すのではなく、必要なサービスだけに私設経路を開けられます。PSA エンドポイントは実 IP を持つため、NSG・セキュリティリスト・ZPR で「どのワークロードがどのサービスに到達できるか」を制御できます
  • テナンシ外への持ち出しを抑止: PSA 経由のアクセスには自テナンシの認証情報が必要で、他テナンシ宛の利用を防げます(データ持ち出し対策)
  • オンプレミスからも利用可能: FastConnect や Site-to-Site VPN で DRG 経由に接続し、オンプレミスの DNS から VCN リゾルバのリスニングエンドポイントへフォワードすれば、オンプレミスのワークロードもインターネットを経由せずに OCI サービスへ到達できます
  • 追加コストなし・高スループット: 利用は無料で、最大 8 Gb/秒(Object Storage は 25 Gb/秒)、フォルトドメイン/可用性ドメイン障害時のフェイルオーバーも組み込まれています
  • Service Gateway と共存可能: 既存の Service Gateway 構成を残したまま、機微なサービスから段階的に PSA へ移行できます

作成手順(コンソール)

前提として、PSA エンドポイントを配置する VCN とサブネットが必要です。手順は次のとおりです。

  1. コンソールで対象の VCN の詳細ページを開き、「Private service access」タブで「Create」を選択します
  2. PSA エンドポイントの名前を入力し、作成先のコンパートメントを確認します
  3. 「Service」セクションで、アクセスを有効にしたい Oracle Services Network(OSN)のサービスを 1 つ選択します(1 エンドポイントにつき 1 サービス)
  4. 「Network」セクションで、エンドポイントを配置するサブネットのコンパートメントとサブネットを選択します
  5. プライベート IPv4 アドレスの割り当て方法を選びます。自動割り当て(サブネットの空き IP から動的に割り当てる ephemeral、または予約済みプライベート IP を使う persistent)か、手動割り当て(アドレスを直接入力、または既存の予約済み IP から選択)を指定できます
  6. (任意)ZPR のセキュリティ属性を付与し、作成を実行します

作成後は、VCN 内のインスタンスから対象サービスの FQDN が PSA のプライベート IP に解決されることを確認します。

# Object Storage の FQDN が PSA のプライベート IP に解決されるか確認
# (リージョンとネームスペースは環境に合わせて読み替え)
nslookup objectstorage.ap-tokyo-1.oraclecloud.com

# 期待値: PSA エンドポイントに割り当てたプライベート IP(例: 10.0.1.10)が返る
# その後は通常どおり CLI / SDK を使うだけで経路がプライベートになる
oci os bucket list --compartment-id <compartment-ocid>

あわせて、PSA エンドポイントを配置したサブネットのセキュリティリスト(または PSA に付けた NSG)で、利用元ワークロードからの HTTPS(443/TCP)受信を許可しておきます。CLI で管理する場合は oci psa private-service-access コマンド群が利用できます。

補足: 対応サービスと注意点

  • 対応サービスの代表例は OCI API、Object Storage、Yum リポジトリ、Container Registry(OCIR)です。最新の対応サービス一覧は公式ドキュメントで確認してください
  • PSA エンドポイントは VCN 単位に作るため、複数の VCN から使う場合はそれぞれの VCN に作成が必要です
  • Private Endpoint は Object Storage や Autonomous Database などサービス側の機能として提供されており、リソースやネームスペースを限定した閉域アクセスが必要な場合はそちらを選択します
