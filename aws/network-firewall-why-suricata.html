AWS Network Firewall はなぜ Suricata を採用したのか

AWS Network Firewall のステートフル検査エンジンが OSS の IPS「Suricata」互換であることは、AWS 公式ドキュメントとプレスリリースに明記されています。ただし「なぜ Snort ではなく Suricata なのか」を AWS が公式に説明した文書は公開されておらず、その意味で採用理由そのものは未確認です。公開されている一次情報から確度高く推測できる背景は、(1) 設計当初からのマルチスレッドアーキテクチャがマネージドなスケールアウト型サービスと相性が良いこと、(2) Snort 互換のルール言語を持ちルール資産をそのまま活かせること、(3) Network Firewall GA(2020 年 11 月)の時点でマルチスレッド対応の Snort 3 がまだ GA していなかったこと、の 3 点です。

まず事実関係 — 公式情報で確認できること

一次情報で確認できるのは次の事実です。

  • AWS Network Firewall は 2020 年 11 月 17 日に GA しました。Amazon の公式プレスリリースは「Network Firewall はステートフル検査にオープンソースの IPS である Suricata を使用する」と明言しています
  • AWS 公式ドキュメントは、ステートフルルールグループを「Suricata compatible(Suricata 互換)」と一貫して表現しており、ルールは Suricata のルール記法の文字列としてそのまま投入できます
  • ドメインリストルールや 5-tuple ルールといった簡易指定も、内部的には Suricata 互換ルールへ変換されて評価されます

つまり「Suricata ベース(互換)である」ことは公式事実ですが、「なぜ Suricata を選んだか」の理由説明は AWS から公開されていません。以下の各節は、公開情報にもとづく合理的な推測として読んでください。

推測 1: マルチスレッド前提の設計がクラウド規模の検査に向く

Suricata は 2007 年に OISF(Open Information Security Foundation)が「Snort の代替」を目標に開発を始めたエンジンで、最初からマルチスレッドを前提に設計されています(初の安定版 1.0 は 2010 年)。一方、長らく主流だった Snort 2 系はシングルスレッド設計で、マルチコアを活かすにはプロセスを複数並べる運用上の工夫が必要でした。

Network Firewall はトラフィック量に応じて自動でスケールするマネージドサービスです。1 インスタンスあたりのスループットを CPU コア数で伸ばせるマルチスレッド型のエンジンは、こうした「性能を横にも縦にも伸ばす」サービス設計と相性が良いと考えられます。AWS 自身も Open Source Blog「Scaling threat prevention on AWS with Suricata」で、EC2 上で Suricata を使った脅威防御をスケールさせる構成を紹介しており、Network Firewall 登場以前から AWS 上での Suricata 活用に取り組んでいたことがうかがえます。

推測 2: 開発当時、マルチスレッドの Snort 3 はまだ存在しなかった

タイミングの問題も見逃せません。Snort のマルチスレッド対応は Snort 3 で実現しましたが、Snort 3 の正式リリース(GA)は 2021 年 1 月 20 日で、Network Firewall の GA(2020 年 11 月 17 日)より後です。マネージドサービスの開発期間を考えると、AWS がエンジンを選定した時点で実用に足る選択肢は「シングルスレッドの Snort 2」か「マルチスレッドで実績を積んだ Suricata」であり、クラウド規模のトラフィックをさばく前提では Suricata が自然な選択だったと考えられます。

推測 3: Snort 互換のルール言語と成熟したエコシステム

Suricata はもともと「Snort と同じ検知ルール言語を共有する」ことを掲げて開発されており、Snort 向けに蓄積された膨大なルール資産(Emerging Threats など)を大きな書き換えなしに利用できます。実際、Amazon のプレスリリースも「顧客はカスタマイズした Snort および Suricata のルール(広く使われている 2 つのオープンソース形式)を実装できる」と両方の名前を挙げています。

つまり Suricata を採用すれば、エンジンとしてはマルチスレッドの性能を得つつ、ルールのエコシステムでは Snort 資産との互換性も確保できる、いわば「両取り」ができます。利用者側から見ても、既存の IDS/IPS 運用で書き慣れたルールをそのまま持ち込めることは、マネージドファイアウォールの採用障壁を下げる大きな要素です。

# Network Firewall にそのまま投入できる Suricata 互換ルールの例
drop tls $HOME_NET any -> $EXTERNAL_NET any \
  (tls.sni; content:"evil.example.com"; startswith; nocase; endswith; \
   msg:"Blocked TLS SNI to evil.example.com"; sid:1000001; rev:1;)

他の候補との比較で整理すると

  • Snort 2: ルール資産は豊富だがシングルスレッド。スケールを求めるマネージドサービスの中核には不利
  • Snort 3: マルチスレッド対応だが、Network Firewall 開発時点で未リリース(GA は 2021 年 1 月)
  • Zeek(旧 Bro): シグネチャ型 IPS ではなくネットワーク解析・モニタリング(NSM)基盤であり、そもそも用途が異なる
  • Suricata: マルチスレッド設計・Snort 互換ルール・GPLv2 の OSS として 10 年の実績(2020 年時点)があり、非営利団体 OISF が開発を主導

注意: 「Suricata そのもの」ではなく「Suricata 互換」

AWS 公式ドキュメントの表現は一貫して「Suricata compatible」です。Network Firewall はマネージドサービスとして Suricata のルール記法と検査動作を提供しますが、素の Suricata と完全に同一ではなく、利用できるキーワードや動作に制限・注意点があります(例: 一部ルールオプションの非対応など)。詳細は公式の「Limitations and caveats」を確認してください。また、本記事の「なぜ採用されたか」の 3 つの理由づけは、冒頭に書いたとおり AWS の公式説明ではなく公開情報からの推測である点に留意してください。
