SageMaker Model Monitor の標準コンテナとカスタムコンテナ(BYOC)の違い

標準(組み込み)コンテナ sagemaker-model-monitor-analyzer は、表形式(tabular)データ専用の分析エンジンです。内部では Apache Spark 上のオープンソースライブラリ Deequ が統計量の計算・制約(constraints)の自動提案・違反チェックを行い、コードを書かずに監視を始められます。一方、テキスト・画像・embedding などの非構造化データや、標準にない独自メトリクスを監視したい場合は、Model Monitor の拡張ポイントを使ってカスタムコンテナ(Bring Your Own Container, BYOC)を Amazon ECR に登録して差し替えます。監視のスケジューリング基盤(SageMaker Processing ジョブ)は両者で共通です。

前提:Model Monitor とは何か、4 つの監視タイプ

Amazon SageMaker Model Monitor は、本番環境にデプロイした機械学習モデルの「健康診断」を定期的に自動実行するサービスです。エンドポイントや Batch Transform に流れ込むデータ(とモデルの予測結果)をキャプチャしておき、あらかじめ作った「基準値(baseline)」と比べて、ズレ(ドリフト)があればアラートを出します。公式ドキュメントでは、次の 4 種類の監視タイプが定義されています。

  • Data Quality:入力データの統計的な性質(欠損、型、分布など)が学習時からズレていないかを監視する
  • Model Quality:モデルの予測と人間が付けた正解ラベル(Ground Truth)を突き合わせ、accuracy や RMSE などの精度指標を監視する
  • Bias Drift:本番の予測に含まれるバイアス(特定属性への偏り)の変化を SageMaker Clarify で監視する
  • Feature Attribution Drift:「どの特徴量が予測にどれだけ効いているか」(SHAP 値による寄与度)のランキングの変化を SageMaker Clarify で監視する

1. 仕組みの違い:標準コンテナは Deequ で何をしているか

標準コンテナは AWS が提供するビルド済みイメージ sagemaker-model-monitor-analyzer です。公式ドキュメントによると、このイメージは Apache Spark 3.3.0 と Deequ 2.0.2 をベースに構築されており、次の 4 つの機能を提供します。

  • 制約の自動提案(constraint suggestion)
  • 統計量の生成(statistics generation)
  • ベースラインに対する制約の検証(constraint validation)
  • Amazon CloudWatch メトリクスの発行

Deequ は Amazon 社内で開発・利用されている「データの単体テスト」のためのオープンソースライブラリで、列ごとの欠損率・データ型・平均・標準偏差などの統計量を大規模データに対して分散計算します。分位数(中央値など)の計算には KLL sketch という、ストリームデータから正確な分位数を効率よく近似するアルゴリズムが使われており、結果は statistics.json に記録されます。

運用の流れは「健康診断の基準値づくり → 定期健診」に似ています。まず学習データなどを入力にベースライン処理ジョブを実行すると、統計量 statistics.json と制約 constraints.json が生成されます。その後、監視スケジュール(MonitoringSchedule)が定期的に SageMaker Processing ジョブを起動し、キャプチャ済みの本番データを同じ方法で分析して基準値と比較し、違反があれば constraint_violations.json に出力して CloudWatch メトリクスを発行します。

カスタムコンテナ(BYOC)の場合も、この「スケジュールに従って Processing ジョブが起動し、S3 のデータを読んでレポートを書き出す」という枠組み自体はそのまま使われます。違うのは中身で、Deequ の代わりに自分で書いた分析コードが実行されます。何をどう計算するか(独自の統計、独自のドリフト判定、LLM の評価など)は完全に自由です。

2. 対応できるデータの種類

公式 FAQ に「Model Monitor が統計・メトリクスを計算できるのは表形式データのみ」「NLP や CV(コンピュータビジョン)のモデルを監視したい場合は、Model Monitor が提供する拡張ポイント経由で独自コンテナを持ち込める」と明記されています。

  • 標準コンテナ:表形式データのみ。入力フォーマットとしてサポートされるのは CSV とフラットな JSON です
  • カスタムコンテナ(BYOC):コンテナ内の処理を自分で書くため、テキスト・画像・音声・embedding ベクトルなど任意のデータを扱えます。公式サンプルには、BYOC で LLM(Llama 2)の回答品質を評価・監視する例もあります

3. 監視タイプ別:標準でどこまでできて、どこから BYOC か

4 つの監視タイプは、いずれも表形式データであれば AWS 提供のビルド済みコンテナだけで完結します。それぞれの標準機能は次のとおりです。

  • Data Quality:標準コンテナ(analyzer)が Deequ で欠損・型・分布の統計を計算し、ベースラインとの差分を検出します。追加コードは不要です
  • Model Quality:同じ標準コンテナが、キャプチャした予測と S3 にアップロードした Ground Truth ラベルをマージして精度指標を計算します。回帰では MAE・MSE・RMSE・R2、二値分類では混同行列にもとづく accuracy・precision・recall・F1 など、多クラス分類では加重版のメトリクスが標準で計算されます
  • Bias Drift:SageMaker Clarify のビルド済みコンテナが、本番予測のバイアスメトリクスを定期計算し、信頼区間がしきい値を超えたら違反として報告します
  • Feature Attribution Drift:SageMaker Clarify が SHAP 値で特徴量の寄与度ランキングを作り、ベースラインとのランキングの一致度を NDCG スコアで比較します。NDCG が 0.90 を下回ると自動でアラートが上がります

カスタムコンテナが必要になるのは、主に次のケースです。

  • 入力が表形式でない(テキスト、画像、embedding、LLM の出力など)
  • 標準メトリクスにない独自の統計・ドリフト検知アルゴリズム・評価ロジックを使いたい
  • 前処理・後処理スクリプトの差し込みでは足りないほど、分析処理そのものを作り替えたい

なお、BYOC の拡張ポイント(コンテナコントラクト)が公式に文書化されているのは Model Monitor(Data Quality 系の汎用監視)側です。Bias Drift と Feature Attribution Drift は Clarify 専用コンテナで動く前提で説明されており、これらの分析を独自ロジックに置き換えたい場合は、汎用の監視スケジュール + カスタムコンテナとして自作する形になります(Clarify コンテナ自体を差し替える公式手順は確認できませんでした)。

4. カスタムコンテナ(BYOC)を自作する大まかな流れ

Model Monitor の監視スケジュールは、裏では SageMaker Processing ジョブを起動します。そのため BYOC は「Processing ジョブのコントラクト(約束事)を守るコンテナ」を作る作業になります。手順の骨子は次のとおりです。

  1. 分析コードを書く:コンテナ起動時に実行されるスクリプトが、環境変数からコンテキストを受け取ってデータを分析します。主な環境変数には、入力データの場所 dataset_source(例: /opt/ml/processing/endpointdata)、出力先 output_path(例: /opt/ml/processing/resultdata)、分析対象期間の start_time / end_time、ベースラインの統計・制約ファイルのパス、publish_cloudwatch_metrics などがあります
  2. レポートを規約どおりに出力する:分析結果は output_path に書き出すと S3 へ自動アップロードされます。statistics.json・constraints.json・constraint_violations.json という標準スキーマに従って出力すると、SageMaker 側の可視化や API で特別扱いされます(この特別扱いは表形式データの場合のみ)。CloudWatch メトリクスを出したい場合は /opt/ml/output/metrics/cloudwatch に規定フォーマットで書き込みます
  3. Docker イメージをビルドして Amazon ECR にプッシュする
  4. 監視スケジュールに独自イメージを指定する:SageMaker Python SDK の汎用 ModelMonitor クラスに image_uri として ECR のイメージを渡し、create_monitoring_schedule() でスケジュールを作成します
# 3. ECR へ登録(リージョン・アカウントはダミー値)
aws ecr create-repository --repository-name my-model-monitor
docker build -t my-model-monitor .
docker tag my-model-monitor:latest \
  123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/my-model-monitor:latest
docker push 123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/my-model-monitor:latest
# 4. SageMaker Python SDK で独自イメージを監視スケジュールに指定
from sagemaker.model_monitor import ModelMonitor

monitor = ModelMonitor(
    role=role,
    image_uri="123456789012.dkr.ecr.ap-northeast-1.amazonaws.com/my-model-monitor:latest",
    instance_count=1,
    instance_type="ml.m5.xlarge",
)
monitor.create_monitoring_schedule(
    endpoint_input=endpoint_name,
    output=MonitoringOutput(source="/opt/ml/processing/resultdata", destination=s3_report_path),
    schedule_cron_expression=CronExpressionGenerator.hourly(),
)

5. 試験対策:標準か BYOC かを見分けるキーワード

AWS 認定 Machine Learning Engineer - Associate 相当の問題では、問題文中のデータ種別とメトリクスの種類に注目すると判断できます。

  • 「表形式 / CSV / JSON」+「欠損・分布・スキーマのドリフト」 → Data Quality の標準コンテナで対応可能。「Deequ」という単語が出たら標準コンテナの内部実装のことです
  • 「accuracy / precision / recall / F1 / RMSE / MAE / R2 の劣化を監視」+「Ground Truth ラベルと突き合わせ」 → Model Quality の標準機能
  • 「バイアスの変化を監視」「特徴量の寄与度(SHAP)の変化を監視」 → SageMaker Clarify のビルド済みコンテナ(Bias Drift / Feature Attribution Drift)。これも自作不要です
  • 「テキスト / 画像 / NLP / CV / embedding / LLM」「独自メトリクス」「カスタムのドリフト検知アルゴリズム」 → 標準コンテナは表形式のみなので BYOC が必要。「extension points(拡張ポイント)」「ECR にプッシュ」「ModelMonitor クラスに image_uri を指定」が解答側のキーワードになります
  • 迷ったら「表形式+標準メトリクス=標準コンテナ、それ以外=BYOC」と覚えるのが近道です。また、軽微なカスタマイズ(入力の変換など)なら、コンテナを作らず前処理・後処理スクリプトで済む場合がある点も押さえておくと安心です

6. 比較表:できること / できないこと

観点 標準(組み込み)コンテナ カスタムコンテナ(BYOC)
対応データ 表形式のみ(CSV / フラット JSON) 任意(テキスト・画像・embedding・LLM 出力など)
分析エンジン Deequ 2.0.2(Apache Spark 3.3.0) 自作コード(言語・ライブラリは自由)
ベースライン(制約)の自動提案 できる(statistics.json / constraints.json を自動生成) 自分で実装が必要
4 つの監視タイプ 表形式データならすべて標準対応(Bias / Feature Attribution は Clarify コンテナ) 独自メトリクス・非表形式データの監視を自由に実装できる
CloudWatch メトリクス発行 自動 可能(/opt/ml/output/metrics/cloudwatch に規定形式で出力)
SageMaker の可視化・API 連携 そのまま利用可能 標準の JSON スキーマに従えば利用可能(表形式の場合のみ)
実装・運用の手間 ほぼゼロ(設定のみ) コンテナの開発・ECR 管理・メンテナンスが必要
