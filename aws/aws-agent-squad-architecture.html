AWS Agent Squad のアーキテクチャ構成とユースケース

Agent Squad は、複数の AI エージェントを束ねて 1 つの会話をさばくための軽量な OSS フレームワークです(旧名 Multi-Agent Orchestrator)。中核は Orchestrator(司令塔)・Classifier(振り分け役)・Agents(専門家)・Storage(会話履歴) の 4 要素で、ユーザー入力を Classifier が解析して最適な専門エージェントへルーティングし、会話コンテキストを保ったまま応答します。Python と TypeScript の両方で提供され、カスタマーサポートや多言語チャットボットなど「1 つの窓口で多様な要求を受ける」用途に向きます。

Agent Squad とは(位置づけと由来)

Agent Squad は AWS Labs 発のオープンソース・マルチエージェントオーケストレーションフレームワークです。もともと Multi-Agent Orchestrator という名前でしたが、Amazon Bedrock の機能である「Multi-Agent Collaboration」との混同を避けるため Agent Squad に改名されました。AWS のサービスではなくライブラリであり、Amazon Bedrock・Amazon Lex・AWS Lambda などと統合できる一方、OpenAI や Anthropic などサードパーティ LLM も扱える点が特徴です。

ポイントは「特定の万能エージェントを 1 つ作る」のではなく、得意分野の異なる複数の専門エージェントを用意し、入力ごとに最適な担当へ振り分けるという設計思想にあります。

4 つの中核コンポーネント

  • Orchestrator(オーケストレーター): システムの司令塔。リクエストを受けると会話履歴の取得 → Classifier 呼び出し → 担当エージェントへの委譲 → 会話の保存、という一連の流れを統括します。エラー処理・フォールバック・コンテキスト伝播も担当します。
  • Classifier(分類器): ユーザー入力と会話履歴、各エージェントの「特性(説明文)」を突き合わせて、どのエージェントが最適かを判定します。実装として Bedrock Classifier と Anthropic Classifier が用意されています(軽量な Claude Haiku 等を判定専用に使う構成が一般的)。
  • Agents(エージェント): 実際に応答を生成する専門家。すべて同じインターフェースを実装するため、Orchestrator からは均一に扱えます(後述の各種ビルトインに加え、独自エージェントも追加可能)。
  • Storage(ストレージ): 会話履歴を保持し、マルチターンの文脈を維持します。試作用の In-Memory と、永続化用の Amazon DynamoDB が標準で選べ、独自実装にも差し替えられます。

リクエスト処理の流れ

1 回の問い合わせは、おおむね次の順序で処理されます。

  1. ユーザー入力を Classifier が解析する
  2. Classifier がエージェントの特性と会話履歴を評価し、担当エージェントを選ぶ
  3. 選ばれたエージェントが入力を処理して応答を生成する
  4. Orchestrator が会話とエージェント履歴を Storage に保存する
  5. 応答をユーザーへ返す(ストリーミング / 非ストリーミングの両対応)

Classifier が適切なエージェントを選べなかった場合は、設定により「デフォルトエージェント(汎用 BedrockLLMAgent)に回す」か「言い換えを促すメッセージ(NO_SELECTED_AGENT_MESSAGE)を返す」かを選べます。

主なビルトインエージェント

  • BedrockLLMAgent: Amazon Bedrock の各モデルを Converse API 経由で利用する基本エージェント。汎用のデフォルトエージェントにも使われます。
  • AmazonBedrockAgent: Bedrock Agents(マネージドのエージェント機能)をそのまま呼び出すエージェント。
  • LexBotAgent: Amazon Lex のボットと連携する会話エージェント。
  • LambdaAgent: AWS Lambda 関数を呼び出して任意のロジックを実行するエージェント。
  • SupervisorAgent: 「agent-as-tools」アーキテクチャを採用し、リードエージェントが複数の専門エージェントを並列で協調させる新しい仕組み。チーム連携・動的な委譲・コンテキスト共有に対応します。

このほか OpenAI / Anthropic などの外部 LLM エージェントもあり、独自エージェントを追加する拡張も可能です。

SupervisorAgent ―「振り分け」から「協調」へ

Classifier による振り分けは「1 リクエスト = 1 担当」が基本です。これに対して SupervisorAgent は、ある「監督役」エージェントが他の専門エージェントをツールとして呼び出し、必要に応じて複数を並列実行して結果を統合します。たとえば「旅行手配で、フライト担当・ホテル担当・天気担当を同時に動かして 1 つの回答にまとめる」といった、より高度なチーム作業に向きます。

代表的なユースケース

  • カスタマーサポート: 注文・返品・技術サポート・課金など、問い合わせ種別ごとに専門エージェントを置き、窓口を一本化する(AWS の公式 Guidance でも customer support が題材)。
  • 多言語・汎用チャットボット: 質問の話題に応じて担当を切り替えつつ、汎用エージェントをフォールバックに置く。
  • 旅行・予約サービス: フライト/宿泊/現地情報など分野別エージェントを SupervisorAgent で協調させる。
  • 業務自動化(EC・社内ヘルプデスク等): LambdaAgent で既存 API・業務ロジックを呼び出し、自然言語の窓口から実処理につなぐ。

導入のしかた(言語とデプロイ)

Python と TypeScript の両方で提供され、必要なコンポーネントだけを入れるモジュール構成です。ローカルでも、AWS Lambda 上でも動かせます。Python での最小インストール例は次のとおりです。

# Python 版(AWS / Anthropic / OpenAI の統合込み)
pip install "agent-squad[aws]"

# TypeScript 版
npm install agent-squad
