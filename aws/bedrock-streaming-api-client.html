Amazon Bedrock ストリーミング API クライアントとは

Amazon Bedrock の「ストリーミング API クライアント」とは、bedrock-runtime エンドポイントに対して InvokeModelWithResponseStream または ConverseStream を呼び出し、モデルが生成するトークンを最終応答を待たずに逐次受け取るクライアントのことです。

レスポンスは HTTP の application/vnd.amazon.eventstream(イベントストリーム)で返り、SDK が messageStart → contentBlockDelta → messageStop → metadata という一連のイベントに分解します。チャットの体感速度(最初の文字が出るまでの時間=TTFT の短縮)を改善するのが主な狙いです。

2 つの API ファミリーと、そのストリーミング版

Bedrock の推論 API には 2 系統があり、それぞれに同期版とストリーミング版があります。

  • Invoke 系: モデル固有のリクエストボディを使う。InvokeModel(同期・単一応答)と InvokeModelWithResponseStream(ストリーミング)。
  • Converse 系: モデル非依存の統一インターフェース。Converse(同期)と ConverseStream(ストリーミング)。1 度書いたコードを別モデルでも使い回せるのが利点で、新規実装では Converse 系が推奨です。

モデルがストリーミングに対応しているかは、GetFoundationModel を呼んで responseStreamingSupported フィールドを見れば判定できます。

イベントストリームの流れ(ConverseStream)

ConverseStream のレスポンスは、決まった順序のイベント列です。contentBlockIndex で各コンテンツブロックを対応付けます。

  • messageStart: メッセージの開始。role(assistant)を含む。1 応答に 1 回。
  • contentBlockStart / contentBlockStop: 各コンテンツブロックの開始・終了。contentBlockIndex 付き。
  • contentBlockDelta: 部分生成データ。text(部分テキスト)、reasoningContent(思考の途中)、toolUse(ツール入力 JSON の断片)のいずれか。
  • messageStop: メッセージ終了。stopReason を含む。1 応答に 1 回。
  • metadata: usage(トークン数)、metrics(レイテンシ)、ガードレールの trace 情報。1 応答に 1 回。

ストリーム中に届きうる例外イベント

ストリーミングでは、接続後の途中でもエラーがイベントとして流れてきます。クライアントはこれらを受信ループ内で個別にハンドリングする必要があります。

  • internalServerException: サーバ内部エラー。
  • modelStreamErrorException: ストリーム処理中にモデル側で発生したエラー。
  • validationException: 入力が不正。
  • throttlingException: スロットリング(レート/クォータ超過)。指数バックオフで再試行する。
  • serviceUnavailableException: サービス一時利用不可。

呼び出し例(boto3 / ConverseStream)

必要な IAM 権限は bedrock:InvokeModelWithResponseStream です(ConverseStream も内部的にこのアクション権限を要求します)。

import boto3

client = boto3.client("bedrock-runtime", region_name="us-east-1")

resp = client.converse_stream(
    modelId="anthropic.claude-sonnet-4-6",  # ストリーミング対応モデル
    messages=[{"role": "user", "content": [{"text": "ストリーミングの利点を 3 行で"}]}],
    inferenceConfig={"maxTokens": 512, "temperature": 0.3},
)

for event in resp["stream"]:
    if "contentBlockDelta" in event:
        delta = event["contentBlockDelta"]["delta"]
        if "text" in delta:
            print(delta["text"], end="", flush=True)   # 逐次表示
    elif "messageStop" in event:
        print("\n[stopReason]", event["messageStop"]["stopReason"])
    elif "metadata" in event:
        print("[usage]", event["metadata"]["usage"])

低レベルの InvokeModelWithResponseStream を使う場合は、各 chunk イベントの bytes がモデル固有形式の JSON 断片になります。Converse 系の方がモデル差を吸収してくれるため、まず ConverseStream を検討するとよいです。

具体的なユースケース

  • チャットボット / アシスタント UI: 生成中のテキストをタイプライター表示し、TTFT を縮めて体感速度を改善する。最も一般的な用途です。
  • 長文生成(記事・要約・コード): 数千トークンの応答でも、ユーザーは数百ミリ秒で読み始められ、離脱を防げます。
  • ツール使用を伴うエージェント: toolUse デルタで「いまどのツールを、どんな引数で呼ぼうとしているか」を途中表示し、進捗を可視化できます。
  • 推論(reasoning)モデルの思考過程の提示: reasoningContent デルタを使い、回答に至る考え方を逐次見せられます。
  • 音声合成パイプライン: 確定したテキスト片を順次 TTS に渡すことで、生成完了を待たずに発話を開始できます。

逆に、応答全体を一括で後段処理する(JSON を完全に受け取ってからパースする等)バッチ的な用途では、ストリーミングの利点は薄く、同期の InvokeModel / Converse の方が実装がシンプルです。
