CloudWatch RUM はどうやってブラウザのデータを CloudWatch へ集めるのか

CloudWatch RUM(Real User Monitoring)は、監視したいページに埋め込む小さな JavaScript Web クライアントが主役です。クライアントはブラウザの 標準 API(PerformanceObserver / エラーハンドラ / fetch・XHR のラップなど)で 実ユーザーのテレメトリを集め、それを PutRumEvents という API リクエストとして リージョンの RUM データプレーンエンドポイント https://dataplane.rum.<region>.amazonaws.com へ送ります。

送信時の認証は、未認証(ゲスト)の Amazon Cognito ID プールから取得した 一時的な AWS 認証情報で SigV4 署名する仕組みです。つまり「ブラウザ上の JS が直接 AWS の API を叩ける」ことが、CloudWatch 側にデータが集まる理由です。

なぜ CloudWatch にデータが集められるのか

RUM の本体は、AppMonitor を作成すると払い出される コードスニペット(Web クライアントのローダ)です。これを HTML に貼り付けると、 ブラウザがページを開いたときにクライアント JS が読み込まれ、ユーザーのセッション内で動作します。 従来のサーバサイド監視と違い、計測コードが実際の利用者のブラウザの中で動くため、 表示速度・JavaScript エラー・通信失敗といった「現場のユーザー体験」を直接取得できます。

集めたデータをクラウドへ届けられるのは、Web クライアントが CloudWatch RUM の API(rum:PutRumEvents)を直接呼び出せるからです。 ブラウザ上の JS が AWS API を安全に呼べるよう、未認証 Cognito ID プールから一時認証情報を借りて リクエストに署名する、という橋渡しが用意されています。この一連の仕組みが 「ブラウザ → CloudWatch」のデータ収集を成立させています。

ブラウザでのデータ収集方法

Web クライアントは「テレメトリ(telemetries)」という単位で、収集対象を有効化します。 代表的なものは次のとおりです。いずれもブラウザの標準 API をフックして取得します。

  • performance … ページロードのタイミング、Core Web Vitals(LCP / FID / CLS)、 リソース取得時間。PerformanceObserver / Navigation Timing から取得
  • errors … 補足されなかった JavaScript エラー(window.onerror 等)
  • http … 失敗/成功した HTTP リクエスト。fetch と XMLHttpRequest を ラップして計測
  • interaction … クリックなどの DOM イベント
  • replay … rrweb によるセッションリプレイ(任意)

収集したイベントはすぐには送らず、いったんクライアント内のキャッシュにためてから バッチ送信します。既定の挙動は次のとおりです。

  • 既定で 5000 ミリ秒(dispatchInterval)ごとに 1 バッチを送信
  • 1 バッチあたり最大 100 イベント(batchLimit)。先に達した方の条件で送信
  • ページを離れる(visibilitychange / pagehide)際は、アンロードへの影響を 最小化するため navigator.sendBeacon を優先し、使えない場合は fetch に フォールバック(useBeacon 既定 true)

サンプリング率を設定すれば、全ユーザーセッションのうち一定割合だけを収集する、といった制御も可能です。

エンドポイントへの接続詳細

送信先(データ収集エンドポイント)の既定値は、AppMonitor を作成したリージョンの RUM データプレーンです。

https://dataplane.rum.<region>.amazonaws.com

プロキシ(例: API Gateway)を挟む場合は、次のように任意のエンドポイント+パスプレフィックスも指定できます。

https://<restapi_id>.execute-api.<region>.amazonaws.com/<stage_name>/

このエンドポイントに対して、Web クライアントは PutRumEvents アクションを呼び出します。 送信先リージョンは AppMonitor のリージョンと一致している必要があります。例えば us-east-1 で作成した AppMonitor に対して us-west-2 の dataplane へ送ると、 認証(署名検証)に失敗します。

認証(なぜブラウザの JS が AWS API を叩けるのか)

ブラウザに長期の AWS アクセスキーを置くわけにはいきません。そこで既定では、 未認証 Amazon Cognito ID プールを使って一時的な認証情報を借りる仕組みになっています。 流れは次のとおりです。

  1. Web クライアントが Cognito ID プールに対して ID を要求し、一時的な AWS 認証情報を取得する (内部的に STS の一時認証情報が払い出される)
  2. その認証情報を localStorage にキャッシュし、PutRumEvents リクエストを SigV4 で署名して dataplane エンドポイントへ送る

この ID プールに紐づく IAM ロールには、対象 AppMonitor に対する rum:PutRumEvents の許可が必要です。最小限のポリシーは次のような形です。

{
  "Effect": "Allow",
  "Action": "rum:PutRumEvents",
  "Resource": "arn:aws:rum:<region>:<account-id>:appmonitor/<app-monitor-name>"
}

認証情報の取得経路には 2 通りの Cognito 認証フローがあります。

  • Enhanced(簡易)authflow … identityPoolId のみを指定した場合に使われる 推奨フロー。GetId で identity ID を得て、GetCredentialsForIdentity で 直接認証情報を受け取る
  • Basic(クラシック)authflow … identityPoolId と guestRoleArn の両方を指定した場合に使われる。Cognito が発行したトークンを STS で認証情報に交換し、指定ロールを assume する

このほか、アプリ側で setAwsCredentials により独自に認証情報を渡す方法や、 プロキシ構成で SigV4 署名を無効化(signing: false)してバックエンドで署名する方法もあります。

送信が失敗するときの主な原因

  • 広告ブロッカー/トラッキング防止により dataplane.rum.* へのビーコンが遮断される
  • IAM ロールに rum:PutRumEvents 許可がない、または Resource が AppMonitor と一致しない
  • エンドポイントのリージョンが AppMonitor のリージョンと不一致
