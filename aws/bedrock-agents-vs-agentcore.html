Amazon Bedrock Agents と AgentCore は別物? 役割の違いと使い分け

はい、別物です。Amazon Bedrock Agents は、Bedrock 内で action group や knowledge base を設定するだけでエージェントを作れる「フルマネージド・構成ベース」の機能で、 プロンプト管理・オーケストレーション・メモリ・監視などを AWS が引き受けます。 一方 Amazon Bedrock AgentCore(2025 年 10 月 GA)は、LangGraph・CrewAI・Strands などの 任意のフレームワークと任意のモデルで作ったエージェントを、安全にデプロイ・運用するための モジュール型インフラ基盤です。前者は「手早く作る」、後者は「自作エージェントを本番で動かす土台」と 役割が分かれており、対立するものではなく併用も可能です。

ひとことで言うと

混乱しやすいのは、両方とも名前に「Bedrock」と「Agent」が付くからです。ですが指している層が違います。 Bedrock Agents は「エージェントの作り方(プロダクト機能)」、AgentCore は 「エージェントの動かし方(運用基盤)」だと捉えると整理できます。

Amazon Bedrock Agents とは

Amazon Bedrock Agents は、アプリケーションに自律的なエージェントを構成ベースで組み込む機能です。 キャパシティのプロビジョニングやインフラ管理、独自コードの記述は不要で、AWS が プロンプトエンジニアリング・メモリ・監視・暗号化・ユーザー権限・API 呼び出しを管理します。

  • action groups: エージェントが呼べる API / Lambda と、ユーザーから引き出すべきパラメータを定義する
  • knowledge bases: 自社データを参照させて回答を補強する(RAG)
  • multi-agent collaboration: 複数エージェントを連携させる

Amazon Bedrock AgentCore とは

Amazon Bedrock AgentCore は、任意のフレームワーク・モデル・プロトコルで構築したエージェントを、 スケーラブルかつ安全にデプロイ・運用するためのプラットフォームです。2025 年 10 月 13 日に 一般提供(GA)が発表されました。各サービスは必要なものだけを単体で(a la carte)採用できます。

  • Runtime: 完全なセッション分離を備えた実行環境。最大 8 時間の長時間ワークロードに対応
  • Memory: セッション内 / 長期のメモリを管理し、過去のやり取りから文脈を提供する
  • Gateway: API・Lambda・MCP サーバーをツールとして安全に公開・接続する
  • Identity: エージェントごとの ID と認証・認可(Cognito / Okta / Entra ID 等と連携)
  • Observability: Amazon CloudWatch ベースで実行のエンドツーエンドな可視化を提供する
  • 加えて、組み込みの Browser(Web 操作)と Code Interpreter(コード実行)ツール

つまり AgentCore は「エージェントの頭脳(推論ロジック)」そのものではなく、それを本番で動かすための 共通部品群を提供します。エージェントのロジックは自分で(任意のフレームワークで)書きます。

どう使い分けるか

  • 手早く始めたい / 運用も AWS に任せたい → Bedrock Agents。構成だけでエージェントが立ち上がる
  • LangGraph や CrewAI など特定フレームワークを使いたい / Bedrock 外のモデルも使いたい / 細かく制御したい → AgentCore。フレームワークは自由に選び、運用基盤だけ借りる
  • 併用も可能。例えばロジックは自作フレームワークで書いて AgentCore Runtime で動かしつつ、 必要な部品(Memory や Gateway)だけ単体採用する、といった構成が取れる

要するに「Bedrock Agents をやめて AgentCore に乗り換える」という二者択一ではなく、 作りたいものの自由度と運用要件で選ぶのが正しい捉え方です。
