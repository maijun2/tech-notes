mTLS(相互 TLS 認証)とは何か

mTLS(mutual TLS / 相互 TLS 認証)は、通常の TLS ではサーバだけが証明書を提示するのに対し、クライアントとサーバの双方が X.509 証明書を提示し、互いに相手の身元を検証する使い方です。サーバが「相手が誰か」を暗号的に確認できるため、パスワードや API キーに頼らないサービス間認証(ゼロトラスト)の基盤として使われます。プロトコルとしては TLS そのもの(TLS 1.3 = RFC 8446)で、双方向認証を有効にした構成を指す呼び方です。

通常の TLS との違い

HTTPS で使う通常の TLS は 片方向認証(one-way authentication)です。クライアント(ブラウザ)はサーバ証明書を検証して「接続先が本物のサーバか」を確認しますが、サーバ側はクライアントが誰かを TLS では確認しません。クライアントの認証はその後の HTTP レイヤ(ログイン ID/パスワード、Cookie、Bearer トークンなど)で別途行います。

mTLS では、ここにクライアント証明書による検証を加えます。サーバは TLS ハンドシェイク中にクライアントへ証明書の提示を要求し、クライアントが提示した証明書が信頼する CA(認証局)で署名されているか、かつクライアントが対応する秘密鍵を持っているか(=その証明書の正当な持ち主か)を検証します。両者がそれぞれ相手を検証するため「相互(mutual)」と呼びます。

ハンドシェイクで何が起きるか

TLS 1.3 では、サーバが CertificateRequest メッセージを送ることでクライアント認証を要求します(RFC 8446 §4.3.2)。要求を受けたクライアントは Certificate と CertificateVerify を返し、後者でハンドシェイクの記録に対する署名を提示することで秘密鍵の所持を証明します。流れを簡略化すると次のとおりです。

双方の証明書検証が成功して初めてセッションが確立します。どちらかの証明書が信頼できない、失効している、秘密鍵を証明できない場合はハンドシェイクが失敗し、アプリケーションのデータは一切流れません。認証が「アプリより前」のトランスポート層で済む点が mTLS の特徴です。

主な用途

  • サービス間(machine-to-machine)認証: マイクロサービスや API 間の通信で、相手サービスの身元を証明書で確認する。人間のログインがない経路に向く。
  • ゼロトラストネットワーク: 「ネットワーク内だから信頼する」をやめ、通信ごとに相互認証する。サービスメッシュ(Istio / Linkerd など)はサイドカー間を自動で mTLS 化する。
  • OAuth 2.0 のクライアント認証: RFC 8705 は mTLS をクライアント認証手段として標準化し、アクセストークンを証明書に紐付ける(certificate-bound token)ことでトークン漏洩時の不正利用を防ぐ。
  • IoT デバイス認証: デバイスごとに証明書を配布し、各デバイスを個別に認証する。

導入時の注意点

mTLS はクライアント証明書という「もう一つの PKI」を運用する必要があります。具体的には次のコストが増えます。

  • 証明書の発行・配布: クライアントごとに証明書と秘密鍵を安全に配布する仕組みが要る。
  • ローテーションと失効: 有効期限切れによる通信断を避ける更新運用と、漏洩時の失効(CRL / OCSP)の手当てが必要。
  • CA の信頼設計: クライアント検証用に専用の内部 CA を立てるなど、サーバ証明書とは別の信頼チェーンを設計することが多い。

このため mTLS は「ブラウザ越しの一般ユーザ認証」よりも、証明書を配れる閉じた相手(自社サービス間、登録済みクライアント、IoT デバイス)に向いています。サービスメッシュやマネージドな仕組みを使うと、証明書のローテーションを自動化でき運用負荷を下げられます。

# 例: openssl でサーバへ mTLS 接続し、クライアント証明書を提示する
openssl s_client -connect api.example.com:443 \
  -cert client.crt -key client.key -CAfile ca.crt

# 例: curl でクライアント証明書を付けてリクエストする
curl --cert client.crt --key client.key --cacert ca.crt \
  https://api.example.com/health
