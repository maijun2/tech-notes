EKS on Fargate と OKE コントロールプレーンのパッチ適用

EKS on Fargate の OS パッチは AWS が自動で適用し、パッチ済みの新しい Fargate ノードへ Pod を再作成(リサイクル)して置き換えます。利用者が OS にログインして手当てすることはできず、代わりに PodDisruptionBudget(PDB) で無停止性を守るのが基本方針です。

OKE のコントロールプレーンは Oracle がフルマネージドで運用します。重大な脆弱性のパッチは Oracle が自動適用し、利用者が起点になるのは Kubernetes マイナー版のアップグレード操作だけです(操作は利用者、実行は Oracle という責任分担)。

EKS on Fargate のパッチ適用

Fargate では 1 Pod = 1 マイクロ VM(ノード)であり、利用者はノードの OS にアクセスできません。そのため Amazon EKS が定期的にノードの OS をパッチし、ノードをリサイクル(再作成)して置き換えることで最新のセキュリティ更新を反映します。すべての Fargate Pod は、クラスタの Kubernetes バージョンに対して EKS が提供する最新のパッチバージョンで起動します。古いパッチの Pod があると、EKS が最新へ更新するためにリサイクルすることがあります。

退避(eviction)の流れ

  • 影響を最小化するため、Pod は Availability Zone(AZ)単位で退避されます。退避には Kubernetes の Eviction API が使われ、その呼び出しは EKS の監査ログに記録されます。
  • 退避が成功すれば、新しい Pod が最新パッチで起動し、追加の対応は不要です。
  • 退避に失敗した場合、EKS は退避に失敗した Pod の詳細を含むイベントをアカウントへ送信します。イベントにはクラスタ名・Pod・Namespace・Fargate プロファイル・予定されている終了時刻(scheduled termination time)などが含まれます。
  • 終了予定時刻になっても退避できない場合、新しいパッチ済み Pod に置き換えるため、既存 Pod は定期的に削除されます。終了までの猶予はパッチの緊急度によって変わります。

影響を抑える設定 — PodDisruptionBudget

リサイクル時に同時停止する Pod 数を抑えるには、アプリごとに PodDisruptionBudget(PDB) を設定して最小可用性を宣言します。これによりパッチ適用は進めつつ、同時ダウン数を制限できます。

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: sample-pdb
spec:
  minAvailable: 2        # または maxUnavailable: 1
  selector:
    matchLabels:
      app: sample

注意点として、minAvailable: 100% のように 100% を指定すると退避できず失敗します。また 1 つの Pod に複数の PDB がマッチすると eviction subresource が非対応でエラーになります。緊急パッチのイベントを受け取ったら、終了予定時刻の前に手動でローリング置き換えを行うこともできます。

クラスタ・アップグレードとの関係

OS パッチとは別に、クラスタの Kubernetes バージョンを上げた場合は Fargate Pod の kubelet もクラスタに合わせて更新する必要があります。Fargate Pod は再作成されて初めて新しいバージョンで起動するため、クラスタ更新後は対象の Deployment 等をローリング再デプロイして Pod を入れ替えます。

OKE コントロールプレーンのパッチ適用

OKE では、コントロールプレーン(Kubernetes API server、etcd、スケジューラなどのマスターコンポーネント)は Oracle がフルマネージドで運用します。利用者はコントロールプレーンのノードに直接アクセスしたり手動でパッチを当てたりはしません。

脆弱性パッチは Oracle が自動適用

OKE が管理するコントロールプレーンのコンポーネントにセキュリティ脆弱性が見つかった場合、影響を受けるすべてのクラスタにパッチを当てるのは Oracle の責任です。利用者の操作は不要で、Oracle が自動的に緩和します。

マイナー版アップグレードは「操作は利用者・実行は Oracle」

一方、Kubernetes のマイナー版アップグレード(例: 1.29 → 1.30)は共有責任です。Oracle は Console / API / CLI などのインターフェースを提供し、アップグレードを開始するのは利用者です。開始後、実際のコントロールプレーンのアップグレードを実行するのは Oracle です。OKE は利用者が明示的に開始しない限り、コントロールプレーンの Kubernetes バージョンを勝手には上げません。

# CLI でコントロールプレーンの Kubernetes 版をアップグレードする例
oci ce cluster update \
  --cluster-id ocid1.cluster.oc1..example \
  --kubernetes-version v1.30.1

Oracle は Kubernetes プロジェクトのパッチ版を検証し、重要なパッチ版のサポートを順次リリースします。新しいパッチ版のサポートが通知されたら、対応するマイナー版のクラスタをできるだけ早く最新パッチ版へアップグレードすることが推奨されます。

ワーカーノード(データプレーン)は別責任

参考までに、データプレーン(ワーカーノード)側は責任分担が異なります。データプレーンに脆弱性が見つかった場合、パッチ済みイメージを提供するのは Oracle、そのイメージでノードを更新するのは利用者です。マネージドノードなら OS イメージ更新やノード置き換え、仮想ノード(Virtual Nodes)なら OKE がスケーリング・パッチ・コントロールプレーンのアップグレードを含む重要な運用を自動化します。

まとめ

  • EKS on Fargate: OS パッチは AWS が自動適用。手当ての方法は「手動パッチ」ではなく「Pod のリサイクルを前提に PDB で無停止性を守る」こと。
  • OKE コントロールプレーン: 脆弱性パッチは Oracle が自動。利用者が触るのは「マイナー版アップグレードの開始」だけで、実行は Oracle が担う。
