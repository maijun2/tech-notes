ACME プロトコルとは — TLS 証明書の自動管理プロトコル

ACME (Automated Certificate Management Environment) は、TLS 証明書の取得・更新・失効を 自動化するための標準プロトコルです。IETF が RFC 8555 として標準化しており、 もともと Let's Encrypt のために開発されましたが、現在は Sectigo を含む多くの商用 CA が対応しています。 ACME を使うことで「証明書の期限切れ」や「手作業による更新漏れ」という運用上の問題を根本から排除できます。

ACME が解決する問題

従来の証明書管理では、担当者が CA の Web ポータルで CSR を生成・提出し、 メールで証明書を受け取り、サーバに手作業でインストールする手順が一般的でした。 証明書の有効期限(多くは 1〜2 年)が近づくたびにこの作業が必要で、対応漏れによるサービス障害が絶えませんでした。

ACME はこの一連の流れを機械間通信(マシン・ツー・マシン)で完全に自動化します。 クライアントソフトウェアが CA の ACME サーバと HTTPS 通信を行い、ドメイン所有の証明・証明書発行・ サーバへのインストールまでを無人で行います。

ACME の仕組み — フロー概要

ACME のやり取りは大きく 4 ステップで構成されます。

チャレンジの種類

ACME ではドメインの所有を証明するために「チャレンジ」を使います。 主な種類は以下の 3 つです。

チャレンジ 方法 適したシナリオ
HTTP-01 http://<domain>/.well-known/acme-challenge/<token> にトークンを配置 ポート 80 が公開されている Web サーバ
DNS-01 _acme-challenge.<domain> に TXT レコードを追加 ワイルドカード証明書、内部サーバ
TLS-ALPN-01 ポート 443 で ALPN 拡張を用いた TLS ハンドシェイクで証明 HTTP 非公開で TLS のみ公開のサービス

技術仕様のポイント

ACME (RFC 8555) の通信は以下の仕様に基づきます。

  • すべての通信は HTTPS で行い、JSON ペイロードを JWS (RFC 7515) で署名します。
  • CA の ACME ディレクトリ URL にアクセスすると、newAccount / newOrder / revokeCert 等のエンドポイント一覧を取得できます。
  • リプレイ攻撃対策として、毎回 newNonce エンドポイントから Nonce を取得してリクエストに含めます。
  • 証明書は DER 形式の HTTPS レスポンスで返され、クライアントが PEM に変換します。

主なクライアントソフトウェア

ACME クライアントは OSS・商用合わせて多数あります。代表的なものを挙げます。

  • Certbot — EFF が開発。Linux サーバで最もよく使われるクライアント。
  • acme.sh — シェルスクリプト製。DNS-01 チャレンジの自動化に強い。
  • Caddy — Web サーバ自体が ACME クライアントを内蔵し、起動時に自動取得・更新。
  • cert-manager — Kubernetes 向け。クラスタ内の証明書を ACME で自動管理。

Sectigo の ACME 対応

Sectigo は自社の証明書管理プラットフォーム Sectigo Certificate Manager (SCM) で ACME プロトコルをサポートしています。 SCM の ACME 機能を使うと、Let's Encrypt と同じ certbot などのクライアントから Sectigo の商用 OV/EV 証明書を自動発行・更新できます。

Let's Encrypt との主な違いは以下のとおりです。

  • 証明書の種類 — Let's Encrypt は DV のみ。Sectigo は DV に加え、組織検証(OV)・拡張検証(EV)も発行可能です(ただし OV/EV は人手による審査があるため完全自動化はできません)。
  • 有効期限 — Let's Encrypt は 90 日固定。Sectigo は最大 398 日の証明書を発行できます(ACME 経由の場合は CA の設定に依存)。
  • エンタープライズ機能 — 部署単位のポリシー適用・監査ログ・インベントリ管理など、大規模組織向けの管理機能があります。

SCM の ACME エンドポイントは組織ごとに払い出され、externalAccountBinding で SCM アカウントと紐付けます。

# certbot で Sectigo SCM の ACME エンドポイントを使う例
certbot certonly \
  --agree-tos \
  --email admin@example.com \
  --preferred-challenges dns-01 \
  --dns-route53 \
  --server https://acme.sectigo.com/v2/OV \
  --eab-kid  <SCM が発行した EAB Key ID> \
  --eab-hmac-key <SCM が発行した EAB HMAC Key> \
  -d "*.example.com"
