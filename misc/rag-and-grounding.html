RAG をグラウンディングと呼ぶのはなぜ?

厳密には「RAG = grounding(グラウンディング)」ではありません。Grounding は LLM の出力を検証可能な外部の事実に結びつける(=接地させる)という目的・概念で、 RAG はその目的を実現する代表的な手法のひとつです。

両者がほぼ同義のように使われるのは、(1) RAG が grounding の最も普及した実装だから、 (2) Google Cloud などのベンダーが「Grounding」という機能名で RAG ベースの仕組みを提供しているから、の 2 点が大きな理由です。

言葉の出どころ ― 認知科学の "symbol grounding"

そもそも grounding(接地・根拠づけ)という語は、生成 AI 固有の言葉ではありません。 認知科学者 Stevan Harnad が 1990 年に提起した 記号接地問題(Symbol Grounding Problem)に由来します。これは 「形式的な記号操作だけで動くシステムが、その記号(=言葉)に実世界の意味を どう結びつけるか」という問いです。Harnad は「中国語の単語を中国語の辞書だけで覚えようとすると、 記号から記号へと回り続けて実体に辿り着けない」という例えでこの問題を説明しました。

LLM はまさにこの「記号(テキスト)だけで訓練されたシステム」です。学習済みパラメータの中の 知識は古くなったり曖昧だったりし、ときに事実無根の文章(hallucination)を生みます。そこで 生成テキストを、検証可能な情報源という「地面」に接地させる営みを grounding と呼ぶようになりました。語のニュアンスがそのまま生成 AI に持ち込まれた形です。

Grounding は「目的」、RAG は「手法」

主要クラウドのドキュメントでも、両者ははっきり区別されています。Google Cloud は grounding を次のように定義しています。

生成 AI における grounding とは、モデルの出力を検証可能な情報源に結びつける能力のこと。 モデルに特定のデータソースへのアクセスを与えると、grounding がその出力をデータに つなぎ留め(tether)、内容を捏造する可能性を減らす。

そして「grounding はしばしば RAG のような手法を使って実現される」と続きます。 つまり包含関係は次のとおりです。

grounding という「目的」を達成する手段は RAG だけではなく、回答への出典リンク付与、 検索ツールや関数呼び出しによる外部参照、(限定的には)ドメインデータでの fine-tuning なども 含まれます。RAG はその中で最も汎用的で実装しやすいため、grounding の代名詞のように 扱われているわけです。

では、なぜ「RAG=grounding」と聞こえる場面が多いのか

呼び方が混ざる最大の要因はベンダーの製品・機能名です。たとえば Google Cloud の Vertex AI / Gemini には「Grounding」という名前の機能があり、その実体は Google 検索や Vertex AI Search から関連情報を取り出して回答に与える ― つまり RAG の仕組みそのものです。利用者から見ると「Grounding をオンにする = RAG が動く」 ように見えるため、両語が同義に感じられます。

AWS の Prescriptive Guidance なども grounding と RAG をセットで解説しており、 「RAG は grounding を実現する代表的アプローチ」という位置づけで一貫しています。 会話のなかで厳密に区別しないまま使われることも多いですが、 本質は「目的(grounding)」と「手段(RAG)」の関係だと押さえておけば混乱しません。

用語の使い分けまとめ

Grounding(グラウンディング)= 目的 / 概念
  └─ 出力を「検証可能な外部の事実」に接地させること
     ・hallucination を減らす
     ・出典(根拠)を提示できるようにする

RAG(Retrieval-Augmented Generation)= 手法 / 実装
  └─ 質問に関連する文書を検索 → プロンプトに文脈として注入 → 生成
     ・grounding を達成する最も普及した方法
     ・grounding の唯一の方法ではない(出典付与・ツール参照なども grounding)
