HNSW と IVF — 近似最近傍探索アルゴリズムの特徴とユースケース

HNSW(Hierarchical Navigable Small World)と IVF(Inverted File)は、どちらもベクトル検索の「近似最近傍探索(ANN)」を高速化するインデックス手法ですが、発想が異なります。HNSW は多層のグラフを貪欲に辿って近い点へ手繰り寄せる手法で、低レイテンシ・高精度が強みです。IVF は事前にベクトルをクラスタ(セル)に分割した転置インデックスを作り、クエリに近い一部のクラスタだけを走査する手法で、省メモリかつ巨大データに向きます。レイテンシと精度を最優先するなら HNSW、メモリ効率や数億件規模のスケールを重視するなら(多くは量子化と組み合わせた)IVF が定番です。

そもそも ANN(近似最近傍探索)とは

埋め込み(embedding)ベクトル同士の距離が近いものを探すのが最近傍探索です。全ベクトルと総当たりで距離を計算する厳密探索(brute force / Flat)は正確ですが、件数 N に比例して遅くなります。そこで多少の取りこぼし(recall < 100%)を許す代わりに桁違いに速くするのが近似最近傍探索(Approximate Nearest Neighbor, ANN)です。HNSW と IVF は、この「速度と精度のトレードオフをどう設計するか」の代表的な 2 つのアプローチです。

HNSW:多層グラフを辿るグラフ系の手法

HNSW は、各ベクトルをノードとし「近いノード同士をエッジで結んだグラフ」を作り、グラフ上を貪欲に辿って近傍を探します。鍵は階層構造です。スキップリストのように、上の層ほどノードが疎(長距離リンク)、下の層ほど密(短距離リンク)になっており、上層で大まかに目的地付近までワープし、下層で細かく詰めます。

探索の流れ

  1. 最上層の固定された入口点(entry point)から開始する。
  2. 現在地の隣接ノードのうち、クエリに最も近いものへ移動する(貪欲探索)。これ以上近づけなくなったら 1 つ下の層へ降りる。
  3. 最下層(Layer 0)では幅優先的に候補を広げ(ef パラメータで探索幅を制御)、上位 k 件を返す。

構築時は新規ノードを各層に挿入し、近い既存ノードと双方向にエッジを張ります(1 ノードあたりの接続数の上限が M)。検索の計算量は対数的(おおよそ O(log N))で、N が増えてもレイテンシが伸びにくいのが大きな利点です。

主なパラメータ

  • M:1 ノードが持つエッジ数の上限。大きいほど精度↑・メモリ↑・構築時間↑。
  • efConstruction:構築時の探索幅。大きいほどグラフ品質↑・構築時間↑。
  • ef(efSearch):検索時の探索幅。大きいほど recall↑・レイテンシ↑。

IVF:転置インデックスで候補を絞り込むクラスタ系の手法

IVF は、事前に全ベクトルを nlist 個のクラスタへ分け(k-means による粗量子化=coarse quantizer)、各クラスタの重心(centroid)に紐づく転置リスト(inverted list)へ所属ベクトルを格納します。空間は重心ごとの Voronoi セルに分割されるイメージです。検索時はクエリに近い nprobe 個のセルだけを開いて、その中のベクトルとだけ距離を計算します。

探索の流れ

  1. クエリと全 nlist 個の重心の距離を計算し、近い順に nprobe 個のセルを選ぶ。
  2. 選んだセルの転置リストに入っているベクトルとだけ距離を計算し、上位 k 件を返す。

走査するのはおおむね nprobe / nlist の割合のベクトルだけなので、データの大部分をスキップできます。nprobe を上げれば recall は上がりますがレイテンシも増える、という調整しやすいトレードオフが特徴です。なお最近傍がセル境界をまたいだ隣のセルにあると取りこぼす(エッジ問題)ため、recall を確保するには nprobe を適切に設定します。

多くは量子化と組み合わせる(IVFPQ など)

IVF 単体(IVFFlat)はベクトルを生のまま保持しますが、巨大データではメモリが課題になります。そこで Product Quantization(PQ) でベクトルを圧縮した IVFPQ がよく使われ、数億〜十億規模を現実的なメモリに収められます(その分、量子化による精度低下はあります)。FAISS では IVF4096,PQ32 のような文字列でインデックスを表現します。

# FAISS での IVFFlat の例(概念コード)
import faiss
d = 768                       # 次元数
nlist = 4096                  # クラスタ数
quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFFlat(quantizer, d, nlist)
index.train(xb)               # k-means で重心を学習(IVF は train が必須)
index.add(xb)                 # ベクトルを転置リストへ登録
index.nprobe = 16             # 検索時に開くセル数(recall と速度の調整つまみ)
D, I = index.search(xq, k=10)

HNSW と IVF は排他ではなく、IVF の粗量子化器に HNSW を使う IVFHNSW のような組み合わせもあります。

ユースケースの目安

HNSW が向くケース

  • RAG やセマンティック検索など、低レイテンシ・高 recall をオンラインで求める用途。
  • データ件数が数百万〜数千万程度で、メモリに乗せられる規模。
  • 頻繁にベクトルを追加するが、全削除・全更新は少ないワークロード。
  • pgvector / Qdrant / Weaviate / Elasticsearch・OpenSearch など多くのベクトル DB の既定インデックス。

IVF(特に IVFPQ)が向くケース

  • 数億〜十億件規模で、メモリ予算が厳しい大規模インデックス。
  • バッチ的な大量検索や、GPU(FAISS)での高スループット処理。
  • 多少の精度低下を許容してでも、コストとスケールを優先したい場合。
  • 速度と精度を nprobe 一つで運用中に動的調整したい場合。

まとめ

「グラフを辿って近づく HNSW」か「クラスタで候補を絞る IVF」か、という発想の違いがすべての特徴に効いてきます。まずはレイテンシ・精度重視なら HNSW、巨大データ・省メモリ重視なら IVF(+PQ)を起点に、recall とレイテンシ、メモリ予算の実測でつまみ(ef / nprobe)を調整するのが実務的です。
